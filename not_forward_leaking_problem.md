# Not Forward Leaking / Non-Anticipative Provenance Problem

Common terminology: data leakage, temporal leakage, look-ahead bias,
fitted-transformer/preprocessing leakage, stacking/out-of-fold leakage,
non-anticipative provenance.

This document is a coordination note for the unresolved training-order and
information-flow correctness problem. It is intentionally not an implementation
plan yet. The goal is to make the issue concrete enough that we can discuss it
with another engineer and then choose a finite implementation milestone.

The project-local phrase "forward leakage" means that an artifact used for a
training, replay, evaluation, or readiness claim was generated from information
that was not admissible at the causal cut of that claim. The earlier concrete
example is same-slice upstream checkpoint leakage: a downstream update consumes
an upstream component checkpoint that has already adapted to the same time
slice, batch, cursor range, or causal chunk.

## Status

The first anchor-v1 enforcement chain now exists for the concrete laundering
path that originally triggered this note:

```text
checkpoint generation influence
  -> forecast_eval
  -> replay_environment
  -> policy_training / PPO execution proof
  -> Marshal readiness gate
```

For that path, Lattice recomputes accepted-anchor influence from generation
manifests and Runtime facts, Runtime emits forecast/replay/policy provenance,
and Marshal requires structured Lattice certificate state instead of accepting a
certificate ref alone.

This does not close the broader design. Anchor-v1 now has canonical
contract-source ownership through `.jkimyei` references to
the `NO_LOOKAHEAD_CONTRACT` block in
`src/config/kikijyeba.protocol.cwu_02v.dsl`, and the first bundle
publishability proof exists for the policy-training readiness path:
`snapshot_bundle_publishability.v1` requires a concrete representation/MDN/policy
generation vector and trained-against compatibility closure instead of loose
latest checkpoints. Anchor-v1 now also requires a scalar label/reward
availability frontier alongside the accepted-anchor influence frontier; both
must be present, inherited or recomputed through parent evidence, and admissible
for the target claim. Anchor-v1 now also binds a scalar embargo/purged-window
policy fingerprint and half-open anchor window for policy-training readiness;
parent influence and label/reward availability must end no later than the
window begin. `causal_provenance_generalization.v1` is the current aggregate
state over those anchor-v1 subproofs: no-lookahead artifact provenance, scalar
label/reward availability, scalar embargo/purged-window admissibility, inline
artifact-production closure, trained-against interface stability, and snapshot
bundle publishability. Full readiness still needs label/reward horizon semantics
beyond this scalar frontier, interval-set/fold embargo generalization,
interface-stability generalization for bundles, and a generalized
artifact-production fact if more artifact families need the same proof.
Existing smoke/research evidence must not be promoted as policy acceptance,
paper-online readiness, deployment readiness, or complete training unless the
current Lattice/Marshal proof gates pass for the exact claim.

## Core Refinement

The deeper issue is not fundamentally "training order." The issue is
**information-flow admissibility**.

The implementation may still use an inverse component update order as one
schedule pattern, but the proof should not be:

```text
Were components updated in the inverse order?
```

The proof should be:

```text
Was every artifact used for this claim generated from an information state that
was admissible for the claim?
```

In compact form:

```text
legal(U, A) := influence(A) intersection forbidden(U) = empty
```

Where:

- `U` is a use or claim: training, replay generation, evaluation, readiness, or
  bundle publication.
- `A` is a consumed artifact: checkpoint, feature cache, forecast, replay
  output, evaluation report, policy training input, or bundle.
- `influence(A)` is the transitive data/model/artifact state that helped
  produce `A`.
- `forbidden(U)` is the information that must not be visible for the target
  slice, atom set, fold, horizon, embargo window, or claim.

For a simple time-slice protocol:

```text
artifact_fit_frontier(A) < target_slice(U)
```

For stacking or out-of-fold training:

```text
artifact_fit_examples(A) intersection target_examples(U) = empty
```

For financial labels with horizon or embargo:

```text
artifact_label_availability_frontier(A) < as_of_cut(target_slice(U))
```

This turns the question from:

```text
Did policy run before MDN before representation?
```

into:

```text
Were the things consumed by policy for slice S generated by a state that had not
already absorbed forbidden information from S?
```

That distinction matters. The inverse order:

```text
policy -> MDN -> representation
```

is only one possible implementation pattern. The fundamental proof is that
policy inputs for slice `S` were generated from admissible prior generations or
other admissible artifacts.

## Terminology Bridge

The broad industry term is **data leakage**. In time-ordered modeling and
financial backtesting, the closest standard terms are **temporal leakage** and
**look-ahead bias**: the model, feature pipeline, or evaluation process uses
information that would not have been available at the decision time.

Several narrower terms may also apply depending on the final mechanism:

- **Fitted-transformer leakage** or **preprocessing leakage** if an upstream
  fitted component, such as representation, is trained on a slice before it
  produces features used by a downstream model on that same slice.
- **Stacking leakage** or **out-of-fold leakage** if one trained model's
  predictions or latent features are used to train another model without
  prior-generation or out-of-fold separation.
- **Target leakage** if the leaked upstream state encodes labels, future
  returns, realized outcomes, or target-derived quantities.

In this repository, the precise mechanism first identified was:

```text
same-slice upstream checkpoint leakage
```

The broader proof should be named closer to:

```text
no_lookahead_artifact_provenance.v1
```

or:

```text
nonanticipative_snapshot_bundle.v1
```

The implementation milestone can still be named:

```text
training_order_lattice_correctness.v1
```

but that milestone should prove information-flow admissibility and bundle
compatibility, not just wall-clock or topological update order.

## Short Summary

Each trainable component's `.jkimyei` references the relevant canonical
order/admissibility contract for anchor-v1. Lattice enforces the contract by
proving non-anticipative artifact provenance. For the first policy-training
readiness path, Lattice also proves snapshot-bundle publishability so a
readiness-grade handoff uses a coherent generation vector instead of loose
latest component checkpoints.

The current v1 shape avoids three divergent copies of the same rule:

- one canonical contract file:
  `src/config/kikijyeba.protocol.cwu_02v.dsl`.
- every relevant `.jkimyei` references the contract id and digest through
  `NO_LOOKAHEAD_CONTRACT_*` and `COMPONENT_ORDER_CONTRACT_*`.
- component-local `.jkimyei` metadata declares `COMPONENT_ROLE`,
  `SERVING_ORDER_INDEX`, `TRAINING_VISIBILITY_POLICY`,
  `GENERATION_LANE_POLICY`, `VALID_FROM_POLICY`, and
  `ARTIFACT_PROVENANCE_POLICY`.
- Runtime carries the policy `.jkimyei` contract refs into policy-training
  evidence.
- Lattice reads these references and proves the admissibility result.
- Marshal obeys Lattice and should not become the proof authority.

The suspected serving dependency remains:

```text
representation -> MDN -> policy
```

The earlier suspected no-forward-leakage update schedule was:

```text
policy -> MDN -> representation
```

That schedule may be useful, but it is not the invariant. The invariant is:

```text
For any use U over target slice/atoms T, every consumed artifact A must have
transitive influence that excludes the information forbidden for T under the
declared protocol.
```

## Representation Pretraining Admissibility

Representation is not a special exemption from the rule. It is admissible or not
according to the same fields as the other fitted components: influence frontier,
generation lane, valid-from frontier, and contract permission.

For a downstream claim over `S = [1800,2050)`:

- A representation trained only on prior data is admissible when its generation
  manifest proves `influence_anchor_end_exclusive_max <= 1800` and its
  `valid_from_anchor` permits use at the downstream causal cut.
- A representation trained on a bootstrap-only historical split can be
  admissible for later slices only when it is explicitly marked
  `generation_lane=bootstrap` and the canonical contract permits that lane.
- A representation trained over the full corpus, including future anchors, is
  not readiness-grade admissible for earlier causal claims inside that corpus.
  Even self-supervised representation learning can leak future regime structure,
  fitted normalization/statistics, missingness patterns, latent clusters, or
  other corpus-wide information.
- A frozen/init or external pretrained representation is admissible only when
  its manifest and contract declare empty or prior influence for the target
  claim.

The crisp rule is:

```text
Representation may be trained first if and only if the produced checkpoint's
generation manifest proves that its influence is admissible for the downstream
claim that consumes it.
```

Research or transductive full-corpus representation pretraining may remain
useful, but it should be marked as smoke/research/transductive and should not be
promotable to causal readiness claims.

## Problem Statement

The concern is non-anticipative provenance failure: a form of temporal data
leakage or look-ahead bias. Depending on implementation details, it may also be
fitted-transformer leakage, stacking leakage, or target leakage.

The simplest example is same-slice upstream checkpoint leakage. If the
representation component is trained on slice `S`, and then MDN or policy
consumes the representation checkpoint that already incorporates `S`, the
downstream component may indirectly observe information that it should not
receive for a claim over `S`.

Similarly, if MDN is trained on `S` before policy, and policy then consumes an
MDN-derived forecast, replay, or policy-training input produced from that
same-slice-updated MDN, the policy may inherit information from a state that
already adapted to `S`.

The leak may not be a direct checkpoint read. It may be laundered through a
derived artifact:

```text
representation checkpoint
  -> MDN forecast artifact
  -> replay artifact
  -> PPO policy training
  -> policy readiness claim
```

By the time policy trains, it might only read `replay_job_dir`. That replay is
still contaminated if it was generated by an MDN or representation state whose
transitive influence includes forbidden information from the target slice.

The intended correctness property is therefore stronger than "do not train on
validation" and stronger than "run components in the right order":

```text
For any claim U, every consumed artifact must have complete provenance, and its
transitive influence must exclude information forbidden for U.
```

Split protection is necessary but not sufficient. We also need artifact DAG
provenance, causal frontiers, bundle compatibility, and claim-scoped
admissibility checks.

## Current Contradiction

The live system currently proves and documents a downstream-policy path:

```text
representation -> MDN -> forecast evaluation -> Environment replay -> policy
```

Observed current behavior:

- Lattice targets make MDN depend on representation.
- Lattice validation evaluation depends on MDN and representation checkpoints.
- Runtime PPO contracts bind policy training to an existing MDN
  evaluation/replay job through `replay_job_dir`.
- Runtime PPO contracts also bind parent forecast/replay evidence.
- Marshal reports `policy_training_artifact_ready` as non-dispatchable
  artifact-readiness, so it does not train policy as a first-class target
  handoff.

That path may be useful for smoke testing. It is not proof of complete
non-anticipative training correctness. In particular, it does not prove that
the replay/evaluation artifacts used for policy training were produced from a
bundle admissible for the target claim.

## Desired End State

The final system should have these properties:

1. Each relevant component `.jkimyei` references one canonical component-stack
   order/admissibility contract, rather than restating the whole rule
   independently.
2. Runtime emits update and artifact-production evidence with enough provenance
   for Lattice to reconstruct transitive influence.
3. Lattice proves non-anticipative artifact provenance for training, replay,
   evaluation, readiness, and bundle-publish claims.
4. Lattice proves bundle compatibility: a serving/training stack is a coherent
   generation vector, not a loose collection of latest checkpoints.
5. Lattice rejects artifacts whose provenance is missing, whose influence
   frontier intersects forbidden target atoms, or whose generation vector is
   incompatible.
6. Marshal refuses readiness-grade dispatch unless the relevant Lattice
   certificate is present and passing.
7. Marshal does not accept loose "latest checkpoint" paths for readiness-grade
   ordered training. It uses Lattice-approved `bundle_id` or
   `generation_vector` references.
8. Research/smoke artifacts can remain inspectable but must be marked
   non-claimable when the proof is absent.
9. The implementation is efficient enough to be usable on real training runs.

The essential proof should not live in docs. It should live in Lattice.

## Working Vocabulary

`component_order`
: Serving or dependency order declared by a canonical contract and referenced
  by component `.jkimyei` metadata. Example:
  `representation -> MDN -> policy`.

`training_update_order`
: An implementation schedule for updates. It may be the inverse of
  `component_order`, but it is not the core proof.

`information-flow admissibility`
: The claim-scoped rule that a consumed artifact is legal only if its
  transitive influence excludes information forbidden for the target claim.

`non-anticipative artifact provenance`
: Provenance that proves an artifact was generated without seeing information
  beyond the allowed causal cut for the claim it supports.

`use` or `claim`
: A concrete consumption context: training, replay generation, evaluation,
  readiness, bundle publication, or promotion.

`influence(A)`
: The transitive data, fitted model state, derived artifacts, and producer
  events that helped produce artifact `A`.

`forbidden(U)`
: The target atoms, slices, folds, horizon labels, embargo intervals, or
  information states that are not admissible for use `U`.

`causal atom`
: The smallest unit Lattice can reason about for leakage. It should eventually
  include source cursor range, decision time, feature observation time, label or
  reward availability time, split role, fold/out-of-fold group, and embargo or
  horizon group.

`slice`
: A named set or interval of causal atoms. For v1, this may be a monotone
  integer chunk, but the design should not be trapped at row-index semantics.

`generation`
: A checkpoint version for one component after some set of updates.

`snapshot bundle`
: A coherent generation vector approved as a unit, for example:

```text
bundle B_k = {
  representation_generation: R_i,
  mdn_generation: M_j,
  policy_generation: P_l,
  compatibility_edges: [
    M_j trained_against R_i,
    P_l trained_against M_j/R_i-derived artifacts
  ],
  valid_from_slice: S_k,
  influence_frontier: ...
}
```

`compatibility edge`
: Evidence that one generation was trained, evaluated, or declared compatible
  with another generation or interface-stability contract.

`fit_frontier`
: A summary of what training atoms a model or artifact has absorbed.

`label_availability_frontier`
: A stricter frontier for financial labels, rewards, or realized returns whose
  availability may lag or overlap the decision-time slice.

`snapshot barrier`
: A rule controlling which checkpoint generations or bundles are visible to
  which artifact-production or training event.

`forward leakage`
: Project-local shorthand for an artifact reading another artifact or
  checkpoint whose influence includes information that should not be visible
  for the claim.

`same-slice upstream checkpoint leakage`
: The first concrete mechanism under review. It is a direct or transitive case
  where an upstream component has already adapted to slice `S` before producing
  inputs for a downstream claim over `S`.

`out-of-fold separation`
: Standard stacking/cross-fitting discipline where downstream training consumes
  upstream predictions or features generated by a model that was not fit on the
  same examples. This may be an admissible alternative if the canonical
  contract declares it.

## Claim-Scoped Rule

The better candidate rule is claim-scoped:

```text
An artifact generated for target slice or atom set T must not be generated by a
model, checkpoint, bundle, or derived artifact whose transitive fitted state
includes information forbidden for T.
```

For v1, Lattice can approximate this with monotone frontiers:

```text
artifact_influence_max_slice < target_min_slice
```

Later versions should support interval sets, folds, purged windows, horizons,
and embargoes.

The earlier local checkpoint rule remains a useful guardrail:

```text
When training component Ci on slice S, any upstream component Cj where j < i
must be read from a generation that does not include training on S.
```

But this is not sufficient, because a leaky checkpoint can be hidden behind:

```text
checkpoint generation
  -> forecast artifact
  -> replay artifact
  -> policy update
  -> policy artifact
  -> readiness claim
```

Lattice must check the artifact DAG, not only direct checkpoint reads.

## Bundle Compatibility

The system should not reason about individual "latest checkpoints" as if they
were independently safe.

A dangerous pattern is:

```text
read latest representation
read latest MDN
read latest policy
```

A safer pattern is:

```text
read bundle B_17
```

where `B_17` is a coherent vector of generation ids that Lattice has approved.

This matters because avoiding leakage can still create a statistical or
interface mismatch. For example:

```text
MDN M7 was trained against representation R6.
Representation R8 is now latest.
Serving stack tries to use R8 + M7.
```

That may not be look-ahead leakage, but it is still not necessarily a coherent
serving stack. Likewise, if policy was trained using replay generated from
`M_old`, serving that policy with `M_new` may be a mismatch unless an explicit
compatibility proof exists.

The real object should be a published snapshot bundle:

```text
bundle_id
generation_vector
compatibility_edges
valid_from_slice
valid_until_slice
influence_frontier_vector
```

Lattice should prove bundle publishability, not merely checkpoint update order.
Marshal should hand around `bundle_id` or `generation_vector` references for
readiness-grade work, not loose checkpoint paths.

## Trainability, Quarantine, Publishability, Claimability

The design should separate these states:

```text
trainable
quarantined
publishable
claimable
```

A component update may be allowed to run experimentally even if its output is
not claimable. For example:

```text
representation trains on slice S
MDN trains on slice S
```

That can be valid as a private future-state update. It becomes forbidden only
if the resulting artifacts are used to generate training inputs, replay,
evaluation, or readiness evidence for `S` itself.

Lattice can therefore say:

```text
This artifact is quarantined for same-slice claims.
It may become admissible for later slices once its valid_from frontier is
satisfied.
```

This keeps research workflows and smoke runs alive while preventing them from
being promoted into complete training or readiness evidence.

## Causal Bundle Pipeline

A better mental model is publish gates over causal cuts:

```text
Given published bundle B_{k-1}

for target slice S_k:

  1. Generate features / forecasts / replay for S_k using B_{k-1}
     or another bundle whose influence frontier is strictly before S_k.

  2. Train or evaluate policy on S_k only from artifacts generated
     by an admissible prior bundle.

  3. Update representation / MDN / policy using S_k as training data,
     but mark resulting generations as not admissible for S_k claims.

  4. Build one or more candidate bundles.

  5. Lattice approves only bundles whose:
       - internal generations are compatible,
       - influence frontiers are known,
       - derived artifacts inherit provenance,
       - valid_from_slice is after the forbidden region.

  6. Marshal may dispatch future readiness-grade work only from
     Lattice-approved bundles.
```

Slogan:

```text
Consume old bundle for this slice.
Train new state from this slice.
Publish new bundle only for future admissible slices.
```

This is a causal bundle pipeline, not simply inverse order.

## Causal Atoms And Frontiers

The slice definition should be backed by causal atoms.

A causal atom should eventually include at least:

```text
atom_id
source_cursor_range
decision_time_or_interval
feature_observed_at
label_or_reward_available_at
split_role
fold_id_or_oof_group
embargo_group_or_horizon_group
```

Then a slice is a named set or interval of causal atoms.

This matters in financial or time-ordered systems because leakage often hides
in label horizons. An example whose decision time is before `S` may still have
a label or realized return that overlaps `S`. A simple rule like:

```text
fit_frontier < slice_start
```

may be insufficient unless `frontier` means label availability frontier, not
just row index.

For v1, we can keep enforcement simple:

```text
fit_cursor_frontier
fit_decision_time_frontier
fit_label_availability_frontier
embargo_policy_digest
```

Implemented anchor-v1 enforces monotone integer anchor frontiers and a single
half-open purged window carried by policy-training readiness evidence. Later
versions should generalize the same concept to interval sets, folds,
out-of-fold groups, horizon groups, and richer embargo policies.

## Bootstrap Problem

The policy needs MDN/representation signals to be useful. If those components
are completely untrained, the policy may learn nothing useful. But if they are
trained first on the same corpus without admissibility boundaries, we may leak.

Possible bootstrap designs:

1. **Bootstrap split**
   Use a small bootstrap-only split to initialize representation/MDN, and mark
   it explicitly. Lattice must prove that later policy-quality or readiness
   claims do not improperly reuse bootstrap evidence.

2. **Previous-bundle bootstrap**
   Policy trains on slice `S` using MDN/representation artifacts generated by a
   bundle whose influence frontier excludes `S`.

3. **Frozen random/init baseline**
   Start policy with weak signals for the first slices, then improve as
   upstream components mature. This is clean but may be too inefficient.

4. **Alternating causal curriculum**
   Train in causal chunks. For each chunk, policy consumes previous-chunk
   bundle outputs, then representation/MDN/policy candidate states update on
   the current chunk and become eligible only for later chunks.

5. **Two-lane evidence**
   Separate research/pretraining artifacts from readiness-grade artifacts. The
   research lane may pretrain representation/MDN, but readiness-grade proof
   only starts once the ordered/admissible bundle protocol begins.

6. **Out-of-fold alternative**
   For target fold `F`, upstream predictions are generated by models trained on
   folds excluding `F`, if the canonical contract declares out-of-fold
   admissibility.

Each option has different proof and performance implications.

## Missing Ranges Exposure Problem

Working name:

```text
missing_ranges_exposure.v1
```

This is a separate problem from no-lookahead leakage:

```text
No-lookahead proves that the learner did not know forbidden information.
Missing-ranges exposure asks whether the learner used enough admissible
information.
```

Anchor-v1 prevents an upstream component from fitting a target range before
producing same-range artifacts for downstream policy training. That is correct,
but it can create an exposure imbalance:

```text
representation / MDN fit range: historical prior range
policy fit range: later target range
```

For example, in the current anchor-v1 profile:

```text
representation train-core: [0,1600)
MDN train-core:            [0,1600)
forecast/replay coverage:  [1800,2050)
policy PPO fit range:      [1800,2050)
policy output valid from:  >=2050
```

The policy does not train on the representation/MDN fit range as policy
experience, and representation/MDN do not fit the policy target range until they
become future-valid generations. If representation/MDN are updated on
`[1800,2050)`, those new checkpoints are not admissible for same-slice policy
claims over `[1800,2050)`; they are candidates for later slices only.

The first exposure question is whether the upstream range is actually as full as
it could be. In the example above, the gap `[1600,1800)` is only a required gap
if it is blocked by forecast horizon, label/reward availability, embargo, split
policy, or another contract rule. If it is not blocked, the cleaner upstream
fit should be closer to:

```text
representation train-core: [0,1800)
MDN train-core:            [0,1800)
forecast/replay coverage:  [1800,2050)
policy PPO fit range:      [1800,2050)
```

or, more generally:

```text
upstream_fit_end <= latest_admissible_training_frontier(target_slice)
```

This is not a proof failure. It is the cost of causal training. The risk is
performance loss from:

- stale upstream features or forecasts on the policy target range,
- reduced policy sample efficiency because policy cannot use every historical
  range under the same upstream state,
- delayed benefit from improved representation/MDN generations,
- short target windows producing noisy PPO updates,
- increased orchestration cost from repeated walk-forward updates and proof
  checks.

The wrong solution is:

```text
fit representation/MDN on S
produce forecast/replay for S from those updated checkpoints
train or certify policy for S
```

That reopens same-slice upstream checkpoint leakage.

The exposure report must not merely say "range X was not used." It should
classify each gap:

```text
usable_but_unused
forbidden_by_no_lookahead
forbidden_by_embargo_or_label_availability
unknown_availability
```

That distinction matters. A required embargo/horizon gap should not be reported
as wasted exposure, and an unknown label/reward availability range should not be
quietly treated as cleanly usable.

## Proposed Future Approach For Missing Exposure

This is not implemented yet. The current recommendation is a hybrid:

```text
1. Expanding walk-forward upstream training
2. Certified replay bank for policy
3. Rolling policy experience over many certified replay chunks
4. Bootstrap lane for cold start
5. Temporal out-of-fold / cross-fit later, after richer label/reward and
   embargo semantics exist
```

### 1. Expanding upstream fit to the latest admissible frontier

For target slice `S = [begin,end)`, compute:

```text
latest_admissible_upstream_fit_end =
  begin - required_gap
```

where `required_gap` comes from horizon, label/reward availability, embargo, or
split policy. For anchor-v1, this may initially be a scalar gap. Later it should
be interval/fold/horizon aware.

Representation and MDN should train up to that frontier, not stop at an older
arbitrary cutoff. If `[1600,1800)` is admissible for target `[1800,2050)`, it is
usable historical exposure and should normally be included in upstream training.

### 2. Certified replay bank

For each historical chunk `S_k`, generate replay from the bundle admissible at
that chunk's causal cut:

```text
find bundle B_{k-1} admissible at begin(S_k)
generate forecast/replay for S_k from B_{k-1}
store replay chunk with its no-lookahead certificate
```

Future manifest shape:

```text
certified_replay_bank_manifest.v1 {
  bank_id
  claim_kind
  no_lookahead_contract_digest
  component_order_contract_digest
  graph_order_fingerprint
  source_cursor_token
  split_policy_fingerprint

  chunks = [
    {
      chunk_id
      coverage_anchor_begin
      coverage_anchor_end_exclusive
      generated_by_bundle_id
      generated_by_generation_vector_digest
      forecast_eval_fact_digest
      replay_environment_fact_digest
      replay_artifact_digest
      no_lookahead_certificate_digest
      provenance_closure_digest
      consumed_generation_vector_digests[]
      influence_anchor_end_exclusive_max
      admissible_at_anchor
      replay_job_dir_digest_or_report_digest
    }
  ]

  coverage_union
  coverage_gaps
  unknown_availability_ranges
  forbidden_ranges
  bank_digest
}
```

Each chunk must pass no-lookahead for its own coverage range. The bank cannot
launder an invalid chunk into policy training.

### 3. Rolling policy experience

Policy training should eventually consume a certified experience set:

```text
policy_experience_set_digest
certified_replay_chunk_digests[]
coverage_union
per_chunk_certificate_digests[]
policy_experience_mode
```

Instead of training policy only on one short target slice, train on a rolling or
expanding set of certified chunks:

```text
policy_experience_set = {
  replay_chunk[S_{k-N}],
  ...,
  replay_chunk[S_k]
}
```

This can improve policy sample efficiency while preserving:

```text
every chunk was generated from a bundle admissible at that chunk's causal cut
```

### 4. Bootstrap lane

Bootstrap remains useful for cold start:

```text
bootstrap range = [0,B)
readiness starts >= B
```

But bootstrap exposure must be typed. An upstream-only bootstrap range does not
automatically count as policy replay exposure. If policy should train on
bootstrap-era data, replay for that era must itself be generated through an
admissible walk-forward or out-of-fold mechanism.

### 5. Temporal out-of-fold / cross-fit later

Cross-fit can recover data efficiency, but it is not a v1 shortcut. Generic
K-fold can still be invalid for causal claims if future periods train a model
used for earlier-period forecasts. Defer this until the system has stronger:

```text
label_reward_availability_frontier
embargo_purged_window_correctness
fold_id / fold_role semantics
claim_kind semantics
```

The likely future target is temporal OOF, not generic random K-fold.

### 6. PPO validity caveat

A certified replay bank can be causally clean while still not being valid for a
strict on-policy PPO claim. Missing-ranges exposure should track:

```text
policy_experience_mode =
  on_policy_current_rollout
  ppo_epoch_reuse
  offline_replay
  behavior_policy_corrected
  auxiliary_pretrain
  research_only
```

If the policy contract treats PPO as strictly on-policy, rolling replay should
remain smoke/research or auxiliary-only unless behavior-policy/logprob evidence
exists:

```text
behavior_policy_generation_digest
action_logprob_digest
importance_weight_policy
replay_collection_policy_digest
```

This is an RL-validity issue, not no-lookahead leakage, but it must be visible
before a rolling replay bank becomes readiness-grade policy training input.

Anchor-v1 does not solve the data-efficiency problem by itself. It only makes
the information boundary explicit and fail-closed. Performance work should
compare chunk sizes, rolling windows, bootstrap starts, and eventually cross-fit
contracts without weakening the no-lookahead proof.

## What Lattice Should Enforce

Lattice needs a first-class proof target or certificate for non-anticipative
artifact provenance and bundle compatibility.

Preferred certificate names:

```text
no_lookahead_artifact_provenance.v1
nonanticipative_snapshot_bundle.v1
```

The certificate should prove four things:

1. **Contract consistency**
   All components reference the same order/slice/visibility contract digest.

2. **Provenance completeness**
   Every consumed artifact has a known producer and transitive influence
   summary.

3. **Non-anticipation**
   For every artifact used for target slice or atom set `T`, its influence
   frontier excludes forbidden information from `T`.

4. **Bundle compatibility**
   The generation vector used together was trained, evaluated, or declared
   compatible as a bundle.

A minimal v1 certificate could look like:

```text
certificate_id
schema_id = no_lookahead_artifact_provenance.v1

order_contract_digest
slice_schema_digest
split_policy_fingerprint
embargo_policy_fingerprint

target_claim_id
target_slice_set
claim_kind = training | replay_generation | evaluation | readiness

consumed_artifacts = [
  {
    artifact_id
    artifact_kind
    producer_event_digest
    producer_bundle_id
    target_coverage
    influence_frontier
    influence_slice_set_digest
  }
]

consumed_bundles = [
  {
    bundle_id
    generation_vector
    compatibility_digest
    valid_from_slice
    valid_until_slice
    influence_frontier_vector
  }
]

lattice_result = pass | fail
failure_reasons = [...]
```

Required proof inputs may include:

- canonical order contract digest.
- component `.jkimyei` contract references and digests.
- slice schema digest.
- split policy fingerprint.
- embargo policy fingerprint.
- target claim id and claim kind.
- target slice or causal atom set.
- consumed artifact ids and kinds.
- producer event digests.
- consumed bundle ids and generation vectors.
- transitive influence frontier summaries.
- compatibility edge digests.
- source cursor tokens.
- graph order fingerprints.
- Runtime job ids.
- parent update and production event digests.

Lattice checks should include:

- component contract references are declared and stable.
- all relevant components reference the same canonical contract digest.
- every consumed artifact has known producer provenance.
- derived artifacts inherit transitive influence from producer bundles and
  input artifacts.
- read checkpoint generations do not include forbidden same-slice or embargoed
  updates.
- artifact influence frontiers do not intersect forbidden target atoms.
- validation/test splits are not training updates.
- candidate generation vectors are compatible or have explicit compatibility
  evidence.
- loose latest-checkpoint stacks are refused for readiness-grade claims.
- artifact facts cannot satisfy complete training or readiness claims unless
  the non-anticipation certificate is satisfied.
- current downstream-policy artifacts remain inspectable but fail the new proof
  until migrated or reclassified.

## Marshal Tool Implications

Marshal should be a bundle/session gate, not a proof engine.

The current Marshal prepare tools are target-centric:

```text
hero.marshal.prepare.train
hero.marshal.prepare.evaluate
```

For readiness-grade ordered training, Marshal should avoid loose checkpoint
paths and latest-path semantics.

Bad operator shape:

```text
train policy with replay_job_dir=/some/latest/replay
```

Better operator shape:

```text
hero.marshal.training_session
```

with a compact argument surface:

```text
mode = plan | dry_run | execute
session_request | session_request_path
required_certificate = no_lookahead_artifact_provenance.v1
include_machine_payload
```

The request file can carry larger schedule details without expanding the public
tool argument count:

```text
claim_kind = policy_training
target_slice_set = ...
component_stack_contract = ...
candidate_bundle_id = ...
```

Then Marshal asks Lattice:

```text
Is this bundle/artifact set admissible for this claim?
```

Marshal should:

- ask Lattice for the non-anticipation and bundle-compatibility certificate.
- materialize or relay Runtime handoff inputs only from approved bundles.
- refuse readiness-grade dispatch when the Lattice certificate is missing or
  failed.
- record receipts linking Runtime update/production jobs to the Lattice proof.
- mark research/smoke execution as `readiness_grade=false`,
  `training_order_certificate=missing`, and `promotion_allowed=false` when the
  certificate is absent.
- not become the proof authority.
- not select best checkpoints or policy quality winners.
- not assemble readiness-grade stacks from loose latest checkpoints.

## Component `.jkimyei` Implications

The component `.jkimyei` references the rule without restating the whole order
independently in every component. Anchor-v1 uses this concrete shape:

```text
NO_LOOKAHEAD_CONTRACT_ID = cwu_02v_no_lookahead_artifact_provenance.anchor_v1
NO_LOOKAHEAD_CONTRACT_DIGEST = cwu_02v_no_lookahead_artifact_provenance.anchor_v1
COMPONENT_ORDER_CONTRACT_ID = cwu_02v_no_lookahead_artifact_provenance.anchor_v1
COMPONENT_ORDER_CONTRACT_DIGEST = cwu_02v_no_lookahead_artifact_provenance.anchor_v1
COMPONENT_ROLE = mdn
SERVING_ORDER_INDEX = 1
TRAINING_VISIBILITY_POLICY = prior_generation_per_slice
GENERATION_LANE_POLICY = readiness_grade_bootstrap_frozen_init_only
VALID_FROM_POLICY = valid_from_anchor_gte_fit_end
ARTIFACT_PROVENANCE_POLICY = transitive_influence_required
```

The canonical contract says:

```text
CONTRACT_ID = cwu_02v_no_lookahead_artifact_provenance.anchor_v1
CERTIFICATE_SCHEMA = no_lookahead_artifact_provenance.v1
FRONTIER_UNIT = accepted_anchor_index
SERVING_ORDER = representation,mdn,policy
VISIBILITY_POLICY = prior_generation_per_slice
DERIVED_ARTIFACT_RULE = inherit_parent_influence
CHECKPOINT_RULE = generation_manifest_required
PUBLISH_RULE = valid_from_anchor_gte_fit_end
BOOTSTRAP_POLICY = explicit_lane_only
RESEARCH_POLICY = smoke_or_research_not_promotable
```

This gives `.jkimyei` ownership without creating multiple subtly divergent
copies of the same rule. Lattice reads and enforces the component-declared
contract reference. Marshal should obey the Lattice result.

## Tests We Need

Negative tests:

- representation trains on slice `S`, then MDN reads that same-slice-updated
  representation for `S`: fail.
- MDN trains on `S`, then policy reads that same-slice-updated MDN or an
  MDN-derived artifact for `S`: fail.
- leaky replay laundering: MDN trains on `S`, forecast artifact for `S` is
  produced using that MDN, replay artifact for `S` is produced from that
  forecast, then policy trains on the replay. Expected: fail, even if policy
  never directly reads the MDN checkpoint.
- incompatible latest checkpoints: MDN `M7` was trained against representation
  `R6`, representation `R8` is now latest, and Marshal tries to build a stack
  from latest `R8 + M7`. Expected: fail or require explicit compatibility
  proof.
- missing artifact producer: policy training consumes `replay_job_dir`, but
  replay has no producer generation vector. Expected: fail closed.
- wall-clock order confusion: policy job runs before MDN job in wall-clock
  time, but policy consumes a replay artifact previously generated by an MDN
  that had already fit `S`. Expected: fail.
- PPO contract binds to a replay job generated from same-slice-updated MDN and
  claims complete training: fail.
- missing generation ids: fail closed.
- missing component order/admissibility contract: fail closed.
- artifact readiness without non-anticipation certificate: cannot become
  complete training or readiness.

Positive tests:

- prior-bundle replay: bundle `B_{k-1}` has influence frontier before `S_k`;
  forecast/replay for `S_k` is generated from `B_{k-1}`; policy trains on
  `S_k` using that replay; outputs are marked valid only for future slices.
  Expected: pass.
- quarantined same-slice upstream update: representation and MDN update on
  `S_k` after policy inputs for `S_k` were already generated from `B_{k-1}`;
  their outputs are not used for `S_k` claims; candidate bundle is marked
  valid from `S_{k+1}` or later. Expected: pass.
- out-of-fold alternative: for target fold `F`, upstream predictions are
  generated by models trained on folds excluding `F`. Expected: pass if the
  contract declares out-of-fold admissibility.
- validation/test splits are excluded from training updates: pass.
- efficient chunked schedule with checkpoint generation manifests: pass.
- current smoke artifacts remain inspectable but not promoted to complete
  training correctness: pass.

Regression tests:

- Lattice target evaluation exposes non-anticipation/provenance deficits.
- Lattice rejects artifact laundering through forecast/replay paths.
- Lattice rejects incompatible generation vectors.
- Marshal refuses readiness-grade training dispatch when Lattice reports the
  certificate missing or failed.
- Marshal accepts only when Lattice reports the bundle/artifact set is
  admissible for the requested claim.
- Runtime evidence is sufficient for Lattice to rebuild the certificate from
  artifacts alone.

## Open Questions

1. What is the exact v1 causal atom: batch, graph-anchor interval, causal
   window, replay episode, or training chunk?

2. Is the real invariant inverse component update order per slice, or is the
   invariant checkpoint/artifact visibility against forbidden target atoms?

3. Is representation allowed to be self-supervised on historical training data
   before ordered/admissible bundle training begins, if that pretraining is
   explicitly separated from readiness evidence?

4. Should MDN be trained before policy on earlier slices but not on the same
   target atoms? If yes, the rule is generation/provenance-based rather than
   absolute order-based.

5. How do we keep training efficient without writing too many full checkpoints?

6. Can we represent checkpoint generations as manifests/update logs instead of
   full materialized checkpoints for every slice?

7. How should resumed training interact with generation ordering and bundle
   compatibility?

8. How should Lattice treat existing artifacts generated by the downstream
   policy path?

9. What does Marshal expose as the operator surface without adding many direct
   arguments?

10. What is the v1 representation for `influence(A)` and `forbidden(U)`?

11. How much set algebra does v1 need? Can monotone frontiers cover the first
    milestone?

12. Which claims are allowed before this is fixed? Likely: smoke evidence,
    read-only report inspection, and non-readiness experimentation. Not
    allowed: complete training, policy acceptance, paper-online readiness,
    deployment readiness.

## Philosophical Core

The blocker is not that the system lacks the right topological sort. The
blocker is that the system does not yet know the difference between:

```text
an artifact that exists
```

and:

```text
an artifact that was admissible to know at the causal cut of the claim it
supports
```

The clean refinement is:

```text
.jkimyei references the contract.
Runtime records artifact/update provenance.
Lattice proves non-anticipation and bundle compatibility.
Marshal dispatches readiness-grade work only from approved bundles.
```

Strong recommendation:

```text
"latest checkpoint" should be illegal in readiness-grade ordered training.
Use Lattice-approved snapshot bundles and artifact provenance certificates
instead.
```
