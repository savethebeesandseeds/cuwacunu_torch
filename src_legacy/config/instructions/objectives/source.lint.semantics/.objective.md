- This objective is `source.lint.semantics`.

Primary objective:
- Inspect the repository source for names whose semantics are misleading,
  stale, overly vague, inconsistent with neighboring vocabulary, or too
  implementation-shaped for the concept they represent.
- Produce a human-facing table of naming improvements with exactly these
  columns, in this order:
  `current | proposal | proposal_alt | rationale`.

Inspection scope:
- Prefer the source-bearing surfaces under `src/`, including C/C++ headers and

- Include variables, functions, methods, structs/classes, enum values,
  namespaces, macros/preprocessor directives, configuration keys, file names,
  and path names.
- Exclude build artifacts, runtime ledgers, secret material, encrypted optim
  payloads, third-party/vendor surfaces, and generated temporary files unless a

Current authority constraint:
- Marshal Hero currently cannot modify repository source code directly.
- Do not launch a Runtime campaign for this objective.
- Do not propose or perform source edits as part of the session. The expected

  objective or human refactor pass.

Semantic review rules:
- Prefer names that reveal domain meaning over names that only reveal storage,
  mechanics, or local control flow.
- Flag names that are accurate only in an old design, names that hide units or
  cardinality, names that overload multiple concepts, and names that collide
  with nearby vocabulary.
- Treat public API names, serialized/configuration names, file/path names, and
  cross-module concepts as higher impact than short-lived local variables.
- Preserve project vocabulary when it is deliberate. Do not Anglicize or
  flatten domain terms such as `iitepi`, `wikimyei`, `jkimyei`, `tsodao`,
  `hashimyei`, or `camahjucunu` merely because they are unfamiliar.
- Distinguish semantic defects from personal taste. If a name is merely compact
  but locally clear, leave it out.
- When a rename would require migration, call out compatibility, docs, or
  validation risk in the rationale instead of presenting it as trivial.

Desired human-facing deliverable:
- Return a Markdown table with exactly these columns:

  | current | proposal | proposal_alt | rationale |
  | --- | --- | --- | --- |

- In `current`, include the exact symbol, directive, file, or path name and a
  concise location hint when useful.
- In `proposal`, give the preferred replacement.
- In `proposal_alt`, give a credible second option, or `keep current` when the
  current name is flawed but every rename carries more risk than value.
- In `rationale`, explain the semantic mismatch and any migration or validation
  concern in one compact sentence.
- Rank rows by expected semantic payoff: public and cross-module confusion
  first, then local implementation clarity.
- Keep the first report focused. Prefer the strongest 25 to 40 findings over a
  noisy exhaustive list, and state if additional lower-confidence candidates
  were intentionally omitted.

Completion rule:
